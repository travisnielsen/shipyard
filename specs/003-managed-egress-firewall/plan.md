# Implementation Plan: Managed Egress via Azure Firewall

**Branch**: `003-managed-egress-firewall` | **Date**: 2026-04-22 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/003-managed-egress-firewall/spec.md`

## Summary

Add a managed egress mode built on Azure Firewall in a dedicated hub VNet peered to the existing Shipyard spoke VNet. The implementation keeps outbound mode mutually exclusive: enabling managed egress removes/excludes NAT Gateway resources and subnet links; disabling managed egress preserves current NAT Gateway behavior and subnet associations. Managed egress enforces default-deny outbound with explicit allow-list support, including DNS-name-based filtering, and validates SKU/capability compatibility.

- `azurerm ~> 4.46`
- `azapi ~> 2.6`

**Storage**: N/A (infrastructure configuration only)

**Target Platform**: Azure virtual network fabric in existing Shipyard resource group/region  
**Project Type**: Infrastructure-as-Code extension to existing Terraform configuration  
**Performance Goals**:
- Preserve outbound connectivity during planned mode switches (SC-004)
- Enforce 100% mode exclusivity after each apply (SC-001)

**Constraints**:
- Must comply with repo file ownership conventions (`networking.tf` for VNet/egress)
- Must preserve NAT mode behavior for existing environments when managed egress is off
- Must support DNS-name based filtering and deny-by-default policy in managed mode
- Must remain private-by-default and least-privilege aligned

**Scale/Scope**:
- One Shipyard environment per deployment
- Managed route control across outbound subnets: `aks_nodes`, `acr_tasks`, `vdi_integration`, `dev_vm`
- One hub VNet + one Azure Firewall deployment per environment in managed mode
