# Implementation Plan: Azure Virtual Desktop Infrastructure

**Branch**: `002-avd-infrastructure` | **Date**: 2026-04-16 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/002-avd-infrastructure/spec.md`

## Summary

Add Azure Virtual Desktop (AVD) to the existing shipyard infrastructure so Entra ID users can sign in and receive a Windows 11 desktop session with VS Code and the Azure CLI pre-installed. The deployment is gated by a `deploy_avd` feature flag and is implemented entirely in Terraform using Azure Verified Modules (AVM). The session host VM (`Standard_D2s_v5`, 8 GB RAM) is placed in the existing `vdi_integration` subnet. Tool installation (VS Code, Azure CLI, AVD agent) is handled by a Custom Script Extension at provisioning time.

- `Azure/avm-res-desktopvirtualization-hostpool/azurerm` `0.4.0`

- New provider: `tls ~> 4.0` (required by VM AVM module for credential generation)

**Storage**: N/A (no application data storage; admin credentials in Azure Key Vault)  
**Testing**: `terraform validate`, `terraform plan` (CI gate per constitution)  
**Target Platform**: Azure (Central US) — existing `azurerm_resource_group.this` and VNet  
**Project Type**: Infrastructure-as-Code (IaC) extension to existing Terraform configuration  
**Performance Goals**: Session host `Standard_D2s_v5` (2 vCPUs, 8 GB RAM); user connection in < 3 min (SC-001)  
**Constraints**: 8 GB RAM per session host (demo), no GPU SKUs, no on-premises AD, Entra ID join only, no public inbound to session hosts, `deploy_avd` feature flag required

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

| Principle | Status | Notes |
|---|---|---|
| I. IaC First | PASS | All AVD resources defined in Terraform; no manual portal steps required |
| II. Private-by-Default Networking | JUSTIFIED EXCEPTION | Session host VMs have no public IPs and accept no inbound connections. AVD Workspace/Host Pool management plane requires `public_network_access_enabled = true` so users can connect via the internet-facing AVD web/desktop client. The RDP session is tunnelled through the Microsoft-managed AVD Gateway — session hosts are never directly reachable from the internet. Exception documented in `research.md` and acknowledged in spec Assumptions. |
| III. Identity-Based Access | PASS | Entra ID join (`AADLoginForWindows` extension); admin password auto-generated and stored in Key Vault via VM AVM module; no stored secrets or passwords in IaC |
| IV. Immutable Workspaces | N/A | This feature provisions VDI infrastructure, not a dev workspace container |
| V. Least-Privilege | PASS | `Desktop Virtualization User` role scoped to the Application Group resource; VM system managed identity has no additional role assignments |
| VI. Simplicity | PASS | Using AVM-managed services throughout; no self-hosted components; no speculative additions |
| Security - Secrets in KV | PASS | Admin password stored in a dedicated AVD Key Vault via VM AVM module |
| Automation - Pinned module versions | PASS | All four AVM module versions pinned exactly |
| Automation - terraform validate + plan on PR | PASS | Covered by existing CI workflow for `infra/` changes |

## Complexity Tracking

| Item | Why Needed | Simpler Alternative Rejected Because |
|---|---|---|
| Public network access on AVD Workspace / Host Pool | Users must connect via internet-facing AVD web client or Windows Desktop client. The AVD connection flow requires the management plane to be reachable over public HTTPS. | Deploying a private endpoint for the AVD management plane requires split-DNS and a private endpoint in the VNet, which then requires users to be on-VPN — prohibitively complex for a demo environment. |
| Terraform version bump to >= 1.10.0 | `avm-res-compute-virtualmachine` v0.20.0 requires `terraform >= 1.10, < 2.0` and the `tls ~> 4.0` provider. This is the current stable AVM VM module and the only version with the non-deprecated `account_credentials` interface. | Pinning an older VM AVM module version brings in deprecated inputs that the AVM team is removing in v1.0.0. |
